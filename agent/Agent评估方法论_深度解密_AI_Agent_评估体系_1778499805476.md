Agent，我们不能只看它“说了什么”（Transcript），更要看它“做了什么”（Outcome）。三、 典型Agent类型的评估策略不同类型的 Agent 需要不同的评估策略。Anthropic 分享了针对三大主流 Agent 的实战经验：1\. 编程类 Agent \(Coding Agents\)核心逻辑：代码好不好，跑跑就知道。评估方法：确定性测试是王道。利用单元测试（Unit 提取的信息源是否真实存在。四、如何考虑Agent评估中的非确定性？智能体的行为在每次运行中都会有差异，这使得评估结果更难解读。每个任务都有一定的成功率，例如：一个生活助手执行「查询天气任务」成功率可能是 90%，而执行「撰写邮件」的成功率可能是 50%而且在一次评估运行中通过的任务，在下一次运行中可能就会失败。而我们测试的目的在于真正衡量智能体完成某个任务的频率（即在所有试验中取得成功的比例）。两个指标可以衡量差别：pass@k：衡量智能体在k 次尝试中获得至少一次正确解的概率。随着 k 的增加，pass@k 得分也会提高——“射门次数”越多，至少成功一次的概率就越高。50% 的 pass@1 得分意味着模型在评估任务中首次尝试就成功完成了一半的任务。pass^k：衡量的是所有 k 次试验都成功的概率。随着k 的增加，pass^k 会下降，因为要求在更多试验中保持一致性难度更大。如果智能体每次试验的成功率为 75%，并且进行了 3 次试验，那么三次试验全部成功的概率为 \(0\.75\)³ ≈ 42%。五、 如何实施AI Agent评估？如何给 Agent 打分？Anthropic 建议组合使用三种评分器：1\.基于代码（Code\-based）：方法：字符串匹配检查、二元测试、静态分析\.\.\.特点：快速、便宜、客观、可复现，但死板、缺乏细微差别。示例场景：检查字符串匹配、代码能否运行、JSON 格式是否正确2\. 基于模型（Model\-based / LLM\-as\-a\-Judge）：方法：基于评分标准的评分、基于参考的评估、自然语言评价\.\.\.特点：灵活、懂语义，但昂贵、有概率性。示例场景：评估“回复是否礼貌”、“总结是否全面”等主观指标。避坑：记得给 LLM 一个“不知道”的选项，防止它在这个环节产生幻觉。3\. 人类（Human）：方法：抽样检查、众包打分、A/B 测试\.\.\.特点：黄金标准、与专家用户判断相符，但最贵、最慢。示例场景：用于校准 LLM 评分器，或者在最后阶段进行抽检。六、避坑指南：别让评估毁了创新Anthropic 在文章中特别强调了几个容易被忽视的陷阱，尤其是关于“过度拟合”的问题。评估结果，而不是路径： 不要在评估中规定 Agent 必须 按什么顺序调用什么工具。Agent 可能会发现你没想到的（甚至更好的）解决路径。案例：Anthropic 发现 Opus 4\.5 模型在某些测试中“失败”了，因为它发现了一个规章漏洞，用更聪明的方法帮用户订到了票。如果我们死板地检查步骤，就会扼杀这种“创造性智能”。能力评估 vs\. 回归测试：能力评估（Capability Evals）：用来爬坡。初始通过率可能很低，目的是发现 Agent 做不到什么。回归测试（Regression Evals）：用来守底。通过率应该接近 100%，一旦掉下来就说明新版本引入了 Bug。策略：当一个能力评估的题目被完美攻克后，它就应该“毕业”进入回归测试库，用来防止未来倒退。
